# Phase 10: Scene Engine and Music Pool - Research

**Researched:** 2026-03-19
**Domain:** Scene generation via GPT-4o, vector-based anti-repetition, music pool management, caption generation
**Confidence:** HIGH (core stack verified in codebase), MEDIUM (empirical threshold calibration required), HIGH (music metadata patterns established)

## Summary

Phase 10 implements the scene generation and music pairing infrastructure for daily video production. The phase depends heavily on established patterns from Phase 9 (Kling integration) and reuses proven services (EmbeddingService, SimilarityService RPC pattern). Core decisions are locked: JSON scene library (version-controlled, not DB), single GPT-4o call returning both prompt and caption, scene embeddings mirroring script embeddings (1536 dims, pgvector), 75-80% similarity threshold (recalibrated from v1.0's 85%), and mood-to-BPM music matching.

**Primary recommendation:** Implement SceneEngine as a parallel service to ScriptGenerationService (both synchronous, costing via same circuit breaker), with scene embeddings stored as a new column on content_history (minimal migration footprint). Dry-run threshold calibration with 20-30 test scene pairs before enabling anti-repetition enforcement in production.

## Architecture Patterns

### SceneEngine Service Architecture
Follows the same synchronous, cost-tracking pattern as ScriptGenerationService:
- Constructor loads scene library from JSON at module level (not per call) — similar to CHARACTER_BIBLE
- `pick_scene()` method returns (scene_prompt: str, caption: str, mood: str, cost_usd: float)
- All OpenAI API calls use synchronous client (required for APScheduler ThreadPoolExecutor)
- Cost returned to caller for circuit breaker tracking
- Rejection constraints loaded from DB and injected into GPT-4o system prompt (same pattern as script generation)

### Pattern 4: Music Matching via Simple Lookup
**What:** Query music_pool WHERE mood = selected_mood AND bpm BETWEEN min AND max
**When to use:** Match scene mood to pre-tagged music track
**Example:**
```python
mood_bpm_map = {
    "playful": {"min": 110, "max": 125},
    "sleepy": {"min": 70, "max": 80},
    "curious": {"min": 90, "max": 100}
}

class MusicMatcher:
    def pick_track(self, mood: str) -> dict:
        bpm_range = mood_bpm_map[mood]
        result = supabase.table("music_pool").select("id, title, file_url").filter(
            "mood", "eq", mood
        ).filter(
            "bpm", "gte", bpm_range["min"]
        ).filter(
            "bpm", "lte", bpm_range["max"]
        ).execute()
        # Random selection from candidates, or weighted by platform availability
        return random.choice(result.data)
```

**Source:** No external library — simple SQL query pattern

### Anti-Patterns to Avoid
- **Don't embed individual words:** Embed the full 2-3 sentence scene prompt, not "cocina" + "olla" separately. Only full prompt captures semantic meaning.
- **Don't hardcode 85% threshold:** v1.0's script threshold is NOT valid for visual scene consistency. Empirical calibration with 20-30 test pairs is mandatory before automation.
- **Don't separate scene embeddings table:** Store on content_history.scene_embedding column alongside existing .embedding. Keeps atomic updates and reduces join complexity.
- **Don't skip seasonal overlay in GPT prompt:** Seasonal dates must be checked server-side and injected before GPT-4o call. No client-side conditional logic for theme selection.
- **Don't query music_pool with LIMIT 1:** Always query by mood + BPM range, then randomly select from candidates. Deterministic "first match" leads to repeated track selection.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Vector similarity search | Custom dot-product calculation | pgvector `<=>` operator via Supabase RPC | pgvector is optimized, indexed, avoids data transfer of 1536-dim vectors |
| Spanish text generation | Custom template system or string formatting | GPT-4o with structured system prompt | GPT-4o handles semantic variation, mood adaptation, and caption formula constraints automatically |
| Music metadata queries | In-memory filtering or custom SQL | SQL WHERE clauses on mood/BPM columns | SQL is fast, indexed, handles platform license flags atomically |
| Scene selection weighting | Weighted random library in Python | JSON weight field + random.choices() with weights | Keeps config in data, not code; easier to adjust without redeploy |
| Rejection constraint injection | Manual list building and prompt concatenation | DB table query + formatted JSON injection (same as ScriptGenerationService) | Reuses proven pattern; scales to many rejections without code changes |

**Key insight:** Phase 10 is primarily integration, not algorithm design. Every core capability (embeddings, similarity, SQL queries, JSON parsing, GPT-4o calls) is proven in existing codebase or standard library. Focus is on wiring services together and tuning the 75-80% threshold via data-driven calibration.

## Common Pitfalls

### Pitfall 1: Hardcoding 85% Similarity Threshold from v1.0
**What goes wrong:** Phase 9 inherits script similarity threshold (85%) and applies it unchanged to scene prompts. Result: in week 2-3, scenes feel repetitive ("cat in kitchen" appears 4 times in 30 days) but similarity check doesn't catch them because semantic distance is ~78%. Or opposite: threshold too strict (90%), AI avoids generating any scene similar to successful ones, resulting in jarring scene transitions.

**Why it happens:** Script similarity (philosophical narrative) is NOT the same as visual scene consistency. A script about "existence" is semantically different from one about "purpose" (85%+ distance), but both can describe the same visual: "cat in living room." Inherited threshold doesn't account for this domain shift.

**How to avoid:**
1. **Pre-deployment calibration (Wave 0):** Create dry-run script that:
   - Generates scene prompts for 20-30 test video descriptions (e.g., videos from past 30 days)
   - Embeds each pair; calculates cosine similarity
   - Outputs similarity matrix + histogram (e.g., "min: 0.62, max: 0.88, median: 0.71")
   - Developer reviews and selects threshold visually: where's the gap between "distinct" and "similar"?
2. **Feature flag for enforcement:** Anti-repetition runs in log-only mode until STATE.md is updated with sign-off
3. **Monitoring:** Track average similarity of daily generated scene to past 7 — should trend 0.65-0.75 if threshold is right
4. **Quarterly re-calibration:** Every 30 days, re-run dry-run script and check drift; update threshold if distribution shifts

**Warning signs:**
- Telegram comments "this cat scene is just like Tuesday's" — visual repetition despite threshold passing
- Similarity check rarely triggers (< 1 rejection per 100 attempts) — threshold too high
- Similarity check always triggers (> 5 rejections per 100 attempts) — threshold too low, AI can't generate variety

### Pitfall 2: Embedding Scene Label, Not Full Prompt
**What goes wrong:** Embed the raw scene combo `{"location": "cocina", "activity": "olla"}` instead of the expanded 2-3 sentence prompt. Result: cosine similarity is shallow (only checks if location/activity words are similar), doesn't capture mood/context nuances. Catch repetition at semantic level, miss it at visual level.

**Why it happens:** Scene label is smaller (cheaper to embed), tempting optimization. But "inspeccionar olla" and "jugar en cocina" both match "cocina" with high cosine distance in raw label space, even though they're visually very different activities.

**How to avoid:** Always embed the full GPT-4o-expanded scene prompt (2-3 sentences), not the raw combo. This is enforced in SceneEngine architecture: `embedding = embed_svc.generate(scene_prompt)` (not scene_label).

**Warning signs:** Similarity check passes for scenes that feel visually identical.

### Pitfall 3: Separate Calls for Scene Prompt and Caption
**What goes wrong:** Call GPT-4o twice (once for scene, once for caption). Cost doubles (~2x API spend per video). Semantic mismatch risk: caption describes "curious cat exploring" but prompt describes "playful cat pouncing" — scene and caption don't align visually.

**Why it happens:** Treating caption generation as an afterthought, separate from scene generation. Tempting to use different model or call independently.

**How to avoid:** Single GPT-4o call returns both in structured JSON response. This is a locked decision from CONTEXT.md. Enforced in SceneEngine.pick_scene() signature: `(prompt, caption, mood, cost)` from one call.

**Warning signs:** API costs are 2x expected; caption doesn't match video mood.

### Pitfall 4: Not Checking License Flags Before Publishing
**What goes wrong:** Music track selected via MusicMatcher for TikTok video, but platform_tiktok flag is False (license only covers YouTube). Video published to TikTok; platform detects unlicensed music via Content ID fingerprinting; copyright strike issued. Or: license_expires_at is in the past; track still selected because query doesn't filter expiry date.

**Why it happens:** License flags stored but not checked in MusicMatcher. Expiry date field exists but not validated at query time. Assumption that "if it's in the pool, it's cleared for use" — false if licenses have expired or weren't validated.

**How to avoid:**
1. **MusicMatcher query enforces license flag:** Always filter `WHERE platform_<target> = TRUE` in query. Never select unlicensed track.
2. **Expiry validation at pool population:** Weekly cron job checks music_pool entries where license_expires_at < NOW(), archives or flags as_archived=true.
3. **Phase 11 enforcement (PUB-01):** Before platform_publish(), validate selected music track's license flag matches target platform.
4. **Logging:** Every music selection logs `track_id, mood, bpm, platform, license_tiktok/youtube/instagram` for audit.

**Warning signs:**
- YouTube videos marked with copyright claim (channel notifications)
- TikTok platform shows "This sound is not available in your region/for your account type"
- music_pool has entries with NULL or past dates in license_expires_at

### Pitfall 5: Scene Rejection Constraints Not Cleared Automatically
**What goes wrong:** Creator rejects a scene on Day 1. Rejection stored. On Day 3, creator hasn't explicitly approved the next 2 videos, so rejection constraint is still active. But AI now avoids a perfectly valid scene that the creator might like. Rejection constraint lingers indefinitely, shrinking the scene pool unnecessarily.

**Why it happens:** Rejection constraints have an expires_at field (from Phase 4), but default expiry is never set, or expiry logic isn't implemented for scenes. Unlike script rejections (which are brief content blocks), scene rejections should auto-expire after ~7 days to reset the scene pool.

**How to avoid:**
1. **Set default expires_at:** When storing scene rejection, set `expires_at = NOW() + INTERVAL '7 days'`
2. **Query with expiry:** Load rejections WHERE `expires_at > NOW()`
3. **Alert on accumulation:** If > 5 active rejections, alert creator to review (pool getting too small)

**Warning signs:** Creator comments "why is the AI avoiding all kitchen scenes?" — stale rejections are over-constraining.

### pgvector Similarity Check for Scenes (Mirroring SimilarityService)
```python
# Source: src/app/services/similarity.py pattern
from supabase import Client

class SimilarityService:
    """Add scene variant of similarity check."""

    def is_too_similar_scene(
        self,
        embedding: list[float],
        threshold: float = 0.78,  # 75-80% range, empirically calibrated
        lookback_days: int = 7
    ) -> bool:
        """
        Returns True if any scene in content_history exceeds the similarity threshold.
        scene_embeddings are stored in content_history.scene_embedding column.
        """
        try:
            result = self._supabase.rpc(
                "check_scene_similarity",  # New SQL function (Phase 10-04)
                {
                    "query_embedding": embedding,
                    "similarity_threshold": threshold,
                    "lookback_days": lookback_days,
                },
            ).execute()
            is_similar = len(result.data) > 0
            if is_similar:
                logger.info(
                    "Scene similarity check: REJECT — %d matching scene(s) above %.0f%% threshold",
                    len(result.data),
                    threshold * 100,
                )
            return is_similar
        except Exception as e:
            logger.error("Scene similarity check failed: %s — defaulting to PASS", e)
            return False  # Fail open: allow generation if check fails
```

### Music Matcher Simple Query
```python
# Source: No existing parallel; straightforward SQL pattern
import random
from supabase import Client

class MusicMatcher:
    """Match scene mood to pre-tagged music tracks."""

    MOOD_BPM_MAP = {
        "playful": {"min": 110, "max": 125},
        "sleepy": {"min": 70, "max": 80},
        "curious": {"min": 90, "max": 100}
    }

    def __init__(self, supabase: Client | None = None) -> None:
        self._supabase = supabase or get_supabase()

    def pick_track(self, mood: str, target_platform: str = "tiktok") -> dict:
        """
        Select a music track matching mood and BPM range.
        target_platform: tiktok | youtube | instagram
        Returns: {"id": uuid, "title": str, "file_url": str, "bpm": int}
        """
        bpm_range = self.MOOD_BPM_MAP[mood]
        platform_flag = f"platform_{target_platform}"

        result = self._supabase.table("music_pool").select(
            "id, title, file_url, bpm"
        ).eq(platform_flag, True).eq(
            "mood", mood
        ).gte("bpm", bpm_range["min"]).lte(
            "bpm", bpm_range["max"]
        ).is_("license_expires_at", None).or_(
            f"license_expires_at.gt.now()"
        ).execute()

        if not result.data:
            raise ValueError(f"No cleared tracks for mood={mood}, platform={target_platform}")

        return random.choice(result.data)
```

**Source:** Straightforward Supabase query pattern; filter syntax from src/app/services
