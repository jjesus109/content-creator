  <name>Task 2: Create services/approval.py — ApprovalService</name>
  <files>src/app/services/approval.py</files>
  <action>
Create `src/app/services/approval.py` with `ApprovalService` class.

**Constructor:** `__init__(self, supabase=None)` — accepts optional supabase Client for testability without live DB. Defaults to `get_supabase()`. Follow VideoStorageService pattern.

**Methods (all synchronous — called from thread pool or PTB callback handlers via asyncio bridge):**

1. `is_already_actioned(self, content_history_id: str) -> bool`
   - Query: `supabase.table("approval_events").select("id").eq("content_history_id", content_history_id).limit(1).execute()`
   - Return `bool(result.data)`
   - Purpose: Idempotency guard — prevents double-processing on accidental double-tap or restart re-delivery

2. `record_approve(self, content_history_id: str) -> None`
   - Insert: `{"content_history_id": content_history_id, "action": "approved"}` into approval_events
   - Cause_code omitted (NULL) — DB constraint allows NULL for approvals

3. `record_reject(self, content_history_id: str, cause_code: str) -> None`
   - Insert: `{"content_history_id": content_history_id, "action": "rejected", "cause_code": cause_code}` into approval_events
   - cause_code values: 'script_error', 'visual_error', 'technical_error', 'off_topic'

4. `get_today_rejection_count(self) -> int`
   - Query approval_events with action='rejected', created_at >= today's date (ISO format), created_at < tomorrow's date
   - Use `date.today().isoformat()` for date boundaries
   - Use Supabase `count="exact"` on select, return `result.count or 0`
   - CRITICAL: read from DB only — never module-level counter (must survive restarts)

5. `write_rejection_constraint(self, cause_code: str) -> None`
   - Map cause_code to pattern_type: 'script_error' -> 'script_class', all others -> 'topic'
   - expires_at = `datetime.now(tz=timezone.utc) + timedelta(days=365)` in ISO format
   - reason_text: `f"Rechazado ({cause_code}) — evitar este patron"`
   - Insert into rejection_constraints table (already exists from migration 0002)

6. `clear_constraints_for_approved_run(self, content_history_id: str) -> None`
   - Query approval_events for today's rejections for ANY content (not just this content_history_id — daily rejections share cause categories)
   - Collect unique cause_codes from rejection rows
   - Map to pattern_types using same mapping as write_rejection_constraint
   - Update rejection_constraints: set expires_at = now() for rows matching those pattern_types
   - Use `.in_("pattern_type", list_of_pattern_types).update({"expires_at": now_iso})`
   - If no rejections found today, return early (nothing to clear)

Imports needed: `from datetime import date, datetime, timedelta, timezone`, `from supabase import Client`, `from app.services.database import get_supabase`

Module docstring: state that all methods are synchronous, run from APScheduler thread context or PTB callback handler (via asyncio bridge), and that DB-backed methods are restart-safe by design.

Add log statements in write_rejection_constraint and clear_constraints_for_approved_run for observability.
