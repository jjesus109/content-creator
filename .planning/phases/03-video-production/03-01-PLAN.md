  <name>Task 1: Migration 0003 — add video columns to content_history</name>
  <files>migrations/0003_video_columns.sql</files>
  <action>
Create `migrations/0003_video_columns.sql` with the following four ALTER TABLE statements. All columns nullable to allow graceful insertion before HeyGen submission:

```sql
-- Phase 3: Video Production columns on content_history
-- heygen_job_id: returned by POST /v2/video/generate; used by poller and webhook to track render
-- video_url: stable Supabase Storage public URL (NEVER the HeyGen signed URL)
-- video_status: lifecycle state; constrained to six valid values including retry sentinel
-- background_url: the dark background image URL used for this render (for consecutive-repeat prevention)

ALTER TABLE content_history
  ADD COLUMN IF NOT EXISTS heygen_job_id    text,
  ADD COLUMN IF NOT EXISTS video_url        text,
  ADD COLUMN IF NOT EXISTS video_status     text
    CHECK (video_status IN (
      'pending_render',
      'pending_render_retry',
      'rendering',
      'processing',
      'ready',
      'failed'
    )),
  ADD COLUMN IF NOT EXISTS background_url   text;
```

The `pending_render_retry` value is the sentinel used by the video poller's retry-once logic: it marks that a first 20-minute timeout has already fired and a resubmission has been made. On a second 20-minute timeout the poller reads this sentinel and transitions to `failed`. This avoids the need for a separate retry_count column.

Migration file naming follows established pattern: `0003_video_columns.sql` — it will be picked up by `run_migrations()` in `db/migrations.py` on the next service start.
