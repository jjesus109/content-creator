Current problematic ffmpeg invocation (lines 209-218):
```python
cmd = [
    "ffmpeg",
    "-ss", "00:00:01",
    "-i", "pipe:0",          # <-- ROOT CAUSE 1: partial file / EOF on piped input
    "-frames:v", "1",
    "-f", "image2",
    "-c:v", "mjpeg",
    "-vf", "scale=320:-1",   # <-- ROOT CAUSE 2: no -pix_fmt / -strict for mjpeg 7.x
    "pipe:1",
]
result = subprocess.run(cmd, input=video_bytes, ...)
```

Root cause 1 — partial file:
  ffmpeg with `-ss` BEFORE `-i` does a fast-seek on the container. When input is
  `pipe:0`, ffmpeg cannot seek and must demux the entire stream first. If the
  video has a moov atom at the end (typical for Kling MP4s), ffmpeg hits EOF
  before it can decode a frame. Fix: write bytes to a NamedTemporaryFile first,
  pass the file path as `-i`, then delete the temp file.

Root cause 2 — mjpeg YUV range:
  ffmpeg 7.x mjpeg encoder rejects yuv420p (limited range) without explicit
  permission. Fix: use `-pix_fmt yuvj420p` (JPEG full-range alias) — no
  `-strict unofficial` needed, but add it anyway for forward compatibility.

```python
# Write video bytes to a named temp file so ffmpeg can seek the container.
# pipe:0 stdin causes partial-file / EOF errors for MP4s with moov at end.
with tempfile.NamedTemporaryFile(suffix=".mp4", delete=False) as tmp:
    tmp.write(video_bytes)
    tmp_path = tmp.name

try:
    cmd = [
        "ffmpeg",
        "-y",                          # overwrite output (defensive)
        "-i", tmp_path,                # real file path — ffmpeg can seek
        "-ss", "00:00:01",             # seek AFTER -i for pipe-safe use
        "-frames:v", "1",
        "-f", "image2",
        "-c:v", "mjpeg",
        "-pix_fmt", "yuvj420p",        # JPEG full-range — fixes ffmpeg 7.x mjpeg rejection
        "-strict", "unofficial",       # belt-and-suspenders for mjpeg encoder
        "-vf", "scale=320:-1",
        "pipe:1",
    ]

    result = subprocess.run(
        cmd,
        capture_output=True,
        timeout=30,
    )
finally:
    os.unlink(tmp_path)
```

Note on `-ss` placement: when using a real file (not pipe:0), place `-ss`
AFTER `-i` for frame-accurate seek (avoids skipping too far). The current
code places `-ss` before `-i` which is optimized for pipe input — flip the
order in the new version.

Keep the existing error handling (returncode check, RuntimeError), BytesIO
wrapping, `.name = "thumbnail.jpg"`, and logging unchanged.
